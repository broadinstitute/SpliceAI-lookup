# Contributing / Developer Guidelines

Thanks for your interest in improving SpliceAI-lookup. This document covers the
repository layout, local development, tests, and how to submit changes. For how to
*use* the hosted APIs or run a prebuilt Docker image, see the [README](README.md).

## Repository layout

- [`index.html`](index.html) — the entire [spliceailookup.broadinstitute.org](https://spliceailookup.broadinstitute.org)
  front-end, hosted via GitHub Pages. ES6 JavaScript with
  [Semantic UI](https://semantic-ui.com) and [jQuery](https://en.wikipedia.org/wiki/JQuery).
  It uses a [custom version of igv.js](https://github.com/bw2/igv.js) that adds track
  types for visualizing SpliceAI & Pangolin scores.
- [`google_cloud_run_services/`](google_cloud_run_services/) — the server-side code for
  the SpliceAI and Pangolin REST APIs, hosted on
  [Google Cloud Run](https://cloud.google.com/run?hl=en). The request logic lives in
  [`server.py`](google_cloud_run_services/server.py) (a [Flask](https://flask.palletsprojects.com/en/3.0.x)
  app); SAI-10k transcript predictions are in
  [`sai10k_predictions.py`](google_cloud_run_services/sai10k_predictions.py). The
  Dockerfiles that build the per-tool / per-genome images are under
  [`google_cloud_run_services/docker/`](google_cloud_run_services/docker/), and
  [`build_and_deploy.py`](google_cloud_run_services/build_and_deploy.py) builds and
  deploys them.
- [`original_flask_app/`](original_flask_app/) — the original Flask web app that
  previously powered the site, kept for reference.

The underlying models are maintained in separate repos:
[bw2/SpliceAI](https://github.com/bw2/SpliceAI) and
[bw2/Pangolin](https://github.com/bw2/Pangolin).

## Local development

### Front-end

`index.html` is a static page. Serve the repo root with any static file server, e.g.:

```
python3 -m http.server 8000
```

then open http://localhost:8000/index.html. By default the dev site routes API calls
to the dev API; edit the API base URLs in `index.html` to point at a local server if
you are working on both ends.

### API server

The easiest way to run the API is the prebuilt Docker images — see
**Running Your Own Local API Server** in the [README](README.md#running-your-own-local-api-server).
The images bundle the reference genome and annotation files each server needs.

To iterate on the server code itself, install the dependencies and run `server.py`
directly:

```
cd google_cloud_run_services
python3 -m pip install -r requirements.txt
```

`server.py` is configured entirely through environment variables and needs at least
`TOOL` and `GENOME_VERSION`:

```
TOOL=spliceai GENOME_VERSION=38 PORT=8080 python3 server.py
```

`TOOL` must be `spliceai` or `pangolin` and `GENOME_VERSION` must be `37` or `38`;
each running server answers requests for only that one tool + genome. The scoring
endpoints also need the model package for the chosen tool (installed separately —
see the comments in [`requirements.txt`](google_cloud_run_services/requirements.txt))
and the reference genome / annotation files that the Dockerfiles copy in from
`docker/ref/` and `docker/<tool>/annotations/` (generated by `build_and_deploy.py`).
Attaching a PostgreSQL cache is optional and documented in the README.

## Running the tests

The tests use Python's `unittest` framework.

```
# API score-consistency snapshot tests (query a running API and compare deltas)
python3 test_api_consistency.py            # add --capture once to record a new baseline
python3 test_sai10k_reference_consistency.py

# server-side unit tests
cd google_cloud_run_services
python3 -m unittest test_sai10k_predictions test_score_consistency
```

The Playwright UI test drives the web UI end-to-end and needs a browser installed
first:

```
pip3 install playwright && python3 -m playwright install chromium
python3 -m unittest test_ui
```

Please run the relevant tests before opening a pull request, and capture a fresh
API-consistency baseline before deploying changes that intentionally alter scores.

## Reporting issues & requesting features

Open a [GitHub issue](https://github.com/broadinstitute/SpliceAI-lookup/issues) using
the provided template. Include the variant, genome version, and parameters when
reporting scoring problems so the result can be reproduced.

## Submitting changes

1. Fork the repo and create a branch for your change.
2. Keep changes focused; update the README / this file when behavior or setup changes.
3. Run the relevant tests (above).
4. Open a pull request describing what changed and why.

## Releasing / deploying

Deployment to Cloud Run is automated by the
[`deploy-on-tag.yml`](.github/workflows/deploy-on-tag.yml) GitHub Actions workflow,
which runs on a pushed git tag. `build_and_deploy.py` builds the Docker images and
updates the Cloud Run services; see its `--help` and the deploy section of the README
for details.
